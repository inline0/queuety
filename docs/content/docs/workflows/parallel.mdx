# Parallel Steps

Workflows can run multiple steps concurrently using `->parallel()`. All handlers in a parallel group execute as independent jobs. The workflow advances to the next step only when every parallel job has completed.

## The parallel() API

Pass an array of step handler classes to `->parallel()`:

```php
use Queuety\Queuety;

$workflow_id = Queuety::workflow('enrich_profile')
    ->then(FetchUserHandler::class)
    ->parallel([
        FetchSocialDataHandler::class,
        FetchOrderHistoryHandler::class,
        FetchSupportTicketsHandler::class,
    ])
    ->then(MergeAndScoreHandler::class)
    ->dispatch(['user_id' => 42]);
```

In this workflow:

1. `FetchUserHandler` runs first (sequential).
2. Three handlers run concurrently: social data, order history, and support tickets.
3. `MergeAndScoreHandler` runs after all three parallel handlers complete.

## How parallel execution works

When the workflow engine reaches a parallel step, it dispatches one job per handler in the group. Each job receives the current workflow state and runs independently. Workers can process these jobs on different threads or machines.

The workflow tracks how many parallel jobs are outstanding. Each time a parallel job completes, its return data is merged into the workflow state. When the last parallel job finishes, the workflow advances to the next step.

## State merging from parallel steps

Each parallel handler returns data that is merged into the workflow state. If two handlers return the same key, the last one to complete wins. To avoid conflicts, use distinct keys:

```php
class FetchSocialDataHandler implements Step {
    public function handle( array $state ): array {
        return [
            'social' => [
                'twitter_followers' => 1200,
                'linkedin_connections' => 500,
            ],
        ];
    }

    public function config(): array {
        return [];
    }
}

class FetchOrderHistoryHandler implements Step {
    public function handle( array $state ): array {
        return [
            'orders' => [
                'total_orders' => 47,
                'total_revenue' => 12500.00,
            ],
        ];
    }

    public function config(): array {
        return [];
    }
}

class FetchSupportTicketsHandler implements Step {
    public function handle( array $state ): array {
        return [
            'support' => [
                'open_tickets' => 2,
                'avg_resolution_hours' => 4.5,
            ],
        ];
    }

    public function config(): array {
        return [];
    }
}
```

After all three complete, the next step receives:

```php
[
    'user_id' => 42,
    // from FetchUserHandler:
    'user_name' => 'Alice',
    // from parallel steps:
    'social'  => ['twitter_followers' => 1200, 'linkedin_connections' => 500],
    'orders'  => ['total_orders' => 47, 'total_revenue' => 12500.00],
    'support' => ['open_tickets' => 2, 'avg_resolution_hours' => 4.5],
]
```

## Mixing sequential and parallel

You can freely mix `->then()` and `->parallel()` in a single workflow:

```php
Queuety::workflow('process_upload')
    ->then(ValidateFileHandler::class)           // Sequential
    ->parallel([                                  // Parallel
        GenerateThumbnailHandler::class,
        ExtractMetadataHandler::class,
        ScanForVirusesHandler::class,
    ])
    ->then(StoreResultsHandler::class)           // Sequential
    ->then(NotifyUserHandler::class)             // Sequential
    ->dispatch(['file_id' => 99]);
```

## Failure handling

If any parallel job fails and exhausts its retries (becomes buried), the entire workflow is marked as `failed`. The state bag preserves the outputs from parallel jobs that completed successfully. When you retry the workflow, only the failed parallel job is re-run.
