# Debug Mode

Queuety includes a debug mode that enables verbose logging in the worker process. This is useful during development and when troubleshooting job failures.

## Enabling debug mode

Define the `QUEUETY_DEBUG` constant in `wp-config.php`:

```php
define('QUEUETY_DEBUG', true);
```

## What debug mode does

When enabled, the worker outputs additional information to STDOUT on each job processing cycle:

- Job ID and handler name when a job is claimed
- Execution time for each job
- State merges for workflow steps
- Stale job recovery events
- Scheduler tick results
- Rate limit checks and skip events

## Checking debug mode programmatically

```php
use Queuety\Config;

if ( Config::debug() ) {
    // Debug mode is active
}
```

## When to use debug mode

**During development.** Enable debug mode to see exactly what the worker is doing as you build and test handlers.

**When diagnosing failures.** If jobs are failing unexpectedly, debug mode shows the execution flow leading up to the failure.

**When testing rate limits.** Debug mode logs when a job is skipped due to rate limiting, making it easier to verify your rate limit configuration.

## Production considerations

Debug mode adds overhead to every worker cycle. Disable it in production by removing or setting the constant to `false`:

```php
define('QUEUETY_DEBUG', false);
```

For production debugging, use the log query commands instead:

```bash
# Check recent failures
wp queuety log --event=failed --limit=20

# Check a specific job's history
wp queuety log --job=123

# Inspect a job's full details including payload
wp queuety inspect 123
```
