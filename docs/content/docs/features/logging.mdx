# Logging

All job and workflow state changes are logged to the `queuety_logs` database table. No log files. Every entry is queryable by job, workflow, handler, event type, and time range.

## Why database logging?

- **Queryable.** Filter by workflow, handler, event type, or time range with SQL.
- **Permanent.** Workflow history survives forever by default. Log retention is independent of job/workflow purging.
- **Atomic.** Log writes are part of the same database transaction as state changes. They are always consistent.
- **No file permissions.** No writable directories needed outside the database.

## What gets logged

Every state transition produces a log entry:

| Event | Description |
|---|---|
| `started` | Job/step execution began |
| `completed` | Job/step finished successfully |
| `failed` | Job/step threw an exception |
| `buried` | Job exceeded max attempts |
| `retried` | Job was scheduled for retry |
| `workflow_started` | Workflow was dispatched |
| `workflow_completed` | All workflow steps finished |
| `workflow_failed` | A workflow step was buried |
| `workflow_paused` | Workflow was paused |
| `workflow_resumed` | Workflow was resumed |

Each log entry includes:

| Column | Description |
|---|---|
| `job_id` | Associated job ID |
| `workflow_id` | Associated workflow ID |
| `step_index` | Step position in workflow |
| `handler` | Handler name or class |
| `queue` | Queue name |
| `event` | Event type |
| `attempt` | Attempt number |
| `duration_ms` | Execution time in milliseconds |
| `memory_peak_kb` | Peak memory usage in KB |
| `error_message` | Error message (for failures) |
| `error_class` | Exception class name (for failures) |
| `error_trace` | Stack trace (for failures) |
| `context` | Additional JSON metadata |
| `created_at` | Timestamp |

## Querying logs via CLI

```bash
# Show all logs from the last 24 hours
wp queuety log

# Filter by job ID
wp queuety log --job=123

# Filter by workflow ID (shows the full workflow history)
wp queuety log --workflow=42

# Filter by handler
wp queuety log --handler=SendEmailHandler

# Filter by event type
wp queuety log --event=failed

# Filter by time
wp queuety log --since="2025-03-28 14:00:00"

# Combine with limit
wp queuety log --handler=SendEmailHandler --limit=100

# JSON output
wp queuety log --workflow=42 --format=json
```

## Log retention

By default, logs are kept forever (`QUEUETY_LOG_RETENTION_DAYS = 0`). To enable automatic pruning:

```php
define('QUEUETY_LOG_RETENTION_DAYS', 90); // Keep 90 days of logs
```

Manual pruning via CLI:

```bash
wp queuety log purge --older-than=90
```

## PHP API

```php
$logger = Queuety::logger();

// Query by job
$entries = $logger->for_job(123);

// Query by workflow
$entries = $logger->for_workflow(42);

// Query by handler
$entries = $logger->for_handler('SendEmailHandler', 50);

// Query by event type
$entries = $logger->for_event(LogEvent::Failed, 50);

// Query by time
$entries = $logger->since(new DateTimeImmutable('-24 hours'), 100);

// Purge old entries
$count = $logger->purge(90); // Delete entries older than 90 days
```

## WordPress hooks

Queuety also fires WordPress action hooks at key lifecycle points. These hooks only fire when WordPress is loaded:

```php
// Job lifecycle
do_action('queuety_job_started', int $job_id, string $handler);
do_action('queuety_job_completed', int $job_id, string $handler, int $duration_ms);
do_action('queuety_job_failed', int $job_id, string $handler, Throwable $exception);
do_action('queuety_job_buried', int $job_id, string $handler, Throwable $exception);

// Workflow lifecycle
do_action('queuety_workflow_started', int $workflow_id, string $name, int $total_steps);
do_action('queuety_workflow_step_completed', int $workflow_id, int $step_index, string $handler, int $duration_ms);
do_action('queuety_workflow_completed', int $workflow_id, string $name, array $final_state);
do_action('queuety_workflow_failed', int $workflow_id, string $name, int $failed_step, Throwable $exception);

// Worker lifecycle
do_action('queuety_worker_started', int $pid);
do_action('queuety_worker_stopped', int $pid, string $reason);
```

Use these hooks to build dashboards, Slack alerts, or external monitoring integrations.
